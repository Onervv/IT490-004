# IT490-004 | System Integration Project

### Overview
This repository contains a simple web‚Äëfronted login demo that integrates a PHP
backend with RabbitMQ.  The goal is to highlight how messages are generated by a
browser, forwarded by PHP to a broker, and consumed by a worker script.

### Project structure

- `public/` ‚Äì web‚Äëfacing document root (set as Apache's `DocumentRoot`).
  Contains:
  - `home.php` ‚Äì gateway/landing page (uses the shared header include; `index.php` now redirects here)
  - `login_page.php` ‚Äì login form page
  - `login.php` ‚Äì AJAX endpoint consumed by `login.js`
  - stub `index.html`/`login.html` (simple redirects to the PHP equivalents)
  - `js/` ‚Äì frontend scripts (e.g. `login.js`)
  - `assets/` ‚Äì static assets like favicons

- `includes/` ‚Äì shared PHP snippets
  - `header.php` ‚Äì common `<head>` section with favicons and Bootstrap
  - `path.inc`, `get_host_info.inc`, `rabbitMQLib.inc` ‚Äì RabbitMQ helper files

- `tools/` ‚Äì command‚Äëline utilities
  - `testRabbitMQClient.php`, `testRabbitMQServer.php`, `testphp.php`,
    `soccerData.php`

- `rpc/` ‚Äì auxiliary RPC scripts called by the CLI tools (e.g. `loginRequest.php`)  * payloads forwarded by `login.php` include the `username` and `password` fields
    (application credentials).  These are independent of the RabbitMQ broker
    user/password defined in `config/testRabbitMQ.ini`.
- `config/` ‚Äì configuration files
  - `testRabbitMQ.ini`, `local.ini`, `host.ini`,
    `testRabbitMQServer.conf`, `testRabbitMQServer.monit`

- `logs/` ‚Äì runtime logs generated by `login.php` (kept outside web root)

This layout keeps the web root minimal and groups similar files together.

1. **Configure Apache**

   Point Apache at the `public/` subdirectory so only web assets are exposed.
   Example virtual‚Äëhost (see `/etc/apache2/sites-available/001-sample.conf`):

   ```apache
   <VirtualHost *:80>
       ServerName localhost
       DocumentRoot /home/onervv/IT490-004/public
       <Directory /home/onervv/IT490-004/public>
           Require all granted
       </Directory>
   </VirtualHost>
   ```

   Reload Apache with `sudo systemctl reload apache2` and verify with
   `apache2ctl -S`.

2. **Run RabbitMQ**

   Ensure a broker is running at the host/port in `testRabbitMQ.ini` and that
   the `test` user/vhost exist.  The built‚Äëin management UI (`http://localhost:15672`)
   can help with this.

3. **Open the login page**

   Navigate to `http://localhost/` (it now serves `index.php`, which redirects to `home.php`). Click the
**Go to Login** button to load `login_page.php` or the **Register** button to
open the registration form (`register.php`). The register page includes simple
client‚Äëside validation for username/password/confirmation. Once you have an
account you can return to the login page, enter credentials and click **Sign
In**.  The message will be forwarded to RabbitMQ.
4. **Watch the logs**

   Debug output is written via `error_log()`:

   * `login.php` logs the received POST and the RabbitMQ request payload
   * `rabbitMQLib.inc` logs connection parameters and the JSON body
   * `testRabbitMQServer.php` logs every incoming request

   Check `/var/log/apache2/error.log` (or the PHP error log configured for your
   environment).

5. **Use CLI tools**

   The command‚Äëline utilities live in `tools/`. Example invocations from the
   project root:

   ```bash
   php tools/testRabbitMQClient.php "hello"
   php rpc/loginRequest.php           # rpc scripts still in rpc/
   ```

### Session & Validation Flow

#### 1. Login Flow
`login.js` ‚Üí `login.php` ‚Üí RabbitMQ ‚Üí remote server

1. User enters credentials on `login_page.php`
2. `login.js` sends POST to `login.php` with `{type: "login", uname, pword}`
3. `login.php` forwards request via RabbitMQ to remote server (`testServer2`)
4. Remote server's `doLogin()`:
   - Validates credentials against DB
   - Creates session: `INSERT INTO sessions(userid, sessionkey_hash, expires_at)`
   - Returns `{status: "ok", session_key: "...", username: "..."}`
5. `login.js` stores in browser via `sessionStorage`
6. Redirects to `dashboard.php`

#### 2. Dashboard Validation
`dashboard.php` ‚Üí RabbitMQ ‚Üí remote server

1. Page loads, JS checks `sessionStorage.getItem('session_key')`
2. If no key ‚Üí redirect to login
3. If key exists ‚Üí POST to `dashboard.php` with `{action: "validate", session_key: "..."}`
4. PHP handler sends via RabbitMQ: `{type: "validate_session", session_key: "..."}`
5. Remote server's `doValidate()`:
   - Hashes the session key: `hash("sha256", $sessionId)`
   - Queries: `SELECT 1 FROM sessions WHERE sessionkey_hash=? AND expires_at > now()`
   - Returns `{status: "ok"}` or `{status: "fail"}`
6. Dashboard JS:
   - `status: "ok"` ‚Üí hide spinner, show dashboard
   - Any failure ‚Üí clear sessionStorage, redirect to login

#### 3. Logout
Currently client-side only: clears `sessionStorage` and redirects to login.
Session remains in DB until expiry (2 hours).

#### Status
- Login creates session in DB
- Session key stored in browser
- Dashboard validates session via backend
- Invalid/expired sessions redirect to login

### Notes

* Messages are plain JSON; the broker only sees the serialized payload.
  Authentication uses the credentials in the `.ini` file, not the
  `username`/`password` fields in the JSON.
* You can rename or reorganise the JS/PHP files as needed ‚Äì update the
  HTML and `require_once()` calls accordingly.
* `js/login.js` contains console debug statements for easier front-end debugging.
