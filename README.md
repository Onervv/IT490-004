# IT490-004 | System Integration Project

### Overview
This repository contains a simple web‑fronted login demo that integrates a PHP
backend with RabbitMQ.  The goal is to highlight how messages are generated by a
browser, forwarded by PHP to a broker, and consumed by a worker script.

### Project structure

The repository is now organised into clean, purpose‑based directories:

- `public/` – web‑facing document root (set as Apache's `DocumentRoot`).
  Contains:
  - `home.php` – gateway/landing page (uses the shared header include; `index.php` now redirects here)
  - `login_page.php` – login form page
  - `login.php` – AJAX endpoint consumed by `login.js`
  - stub `index.html`/`login.html` (simple redirects to the PHP equivalents)
  - `js/` – frontend scripts (e.g. `login.js`)
  - `assets/` – static assets like favicons

- `includes/` – shared PHP snippets
  - `header.php` – common `<head>` section with favicons and Bootstrap
  - `path.inc`, `get_host_info.inc`, `rabbitMQLib.inc` – RabbitMQ helper files

- `tools/` – command‑line utilities
  - `testRabbitMQClient.php`, `testRabbitMQServer.php`, `testphp.php`,
    `soccerData.php`

- `rpc/` – auxiliary RPC scripts called by the CLI tools (e.g. `loginRequest.php`)  * payloads forwarded by `login.php` include the `username` and `password` fields
    (application credentials).  These are independent of the RabbitMQ broker
    user/password defined in `config/testRabbitMQ.ini`.
- `config/` – configuration files
  - `testRabbitMQ.ini`, `local.ini`, `host.ini`,
    `testRabbitMQServer.conf`, `testRabbitMQServer.monit`

- `logs/` – runtime logs generated by `login.php` (kept outside web root)

This layout keeps the web root minimal and groups similar files together.

1. **Configure Apache**

   Point Apache at the `public/` subdirectory so only web assets are exposed.
   Example virtual‑host (see `/etc/apache2/sites-available/001-sample.conf`):

   ```apache
   <VirtualHost *:80>
       ServerName localhost
       DocumentRoot /home/onervv/IT490-004/public
       <Directory /home/onervv/IT490-004/public>
           Require all granted
       </Directory>
   </VirtualHost>
   ```

   Reload Apache with `sudo systemctl reload apache2` and verify with
   `apache2ctl -S`.

2. **Run RabbitMQ**

   Ensure a broker is running at the host/port in `testRabbitMQ.ini` and that
   the `test` user/vhost exist.  The built‑in management UI (`http://localhost:15672`)
   can help with this.

3. **Open the login page**

   Navigate to `http://localhost/` (it now serves `index.php`, which redirects to `home.php`). Click the
**Go to Login** button to load `login_page.php` or the **Register** button to
open the registration form (`register.php`). The register page includes simple
client‑side validation for username/password/confirmation. Once you have an
account you can return to the login page, enter credentials and click **Sign
In**.  The message will be forwarded to RabbitMQ.
4. **Watch the logs**

   Debug output is written via `error_log()`:

   * `login.php` logs the received POST and the RabbitMQ request payload
   * `rabbitMQLib.inc` logs connection parameters and the JSON body
   * `testRabbitMQServer.php` logs every incoming request

   Check `/var/log/apache2/error.log` (or the PHP error log configured for your
   environment).

5. **Use CLI tools**

   The command‑line utilities live in `tools/`. Example invocations from the
   project root:

   ```bash
   php tools/testRabbitMQClient.php "hello"
   php rpc/loginRequest.php           # rpc scripts still in rpc/
   ```

### Notes

* Messages themselves are plain JSON; the broker only sees the serialized
  payload.  Authentication uses the credentials in the `.ini` file, not the
  `username`/`password` fields in the JSON.
* You can rename or reorganise the JS/PHP files as you like – just update the
  HTML and `require_once()` calls accordingly.
* The `js/login.js` file contains console debug statements to make front‑end
  behaviour easier to follow.
